# Network Mechanics

This page covers the technical systems underlying hacking gameplay: how networks are structured, how you move through them, and how security systems try to stop you.

## Network Architecture

Networks in IDApTIK use **simplified realistic networking**. Complex enough to feel authentic, simple enough to be fun.

### Local Networks
Local networks use a **star topology**—all devices connect to a central router or hub:

```
        [Camera]
            |
[Laptop]---[Router]---[Server]
            |
      [Workstation]
```

This keeps network layouts readable while allowing meaningful routing decisions.

### Subnets and Zones
Networks are divided into subnets that create logical boundaries:

- **LAN** (e.g., 192.168.1.x) – Office workstations, printers, personal devices
- **Server VLAN** (e.g., 10.0.0.x) – Internal servers, databases, sensitive systems
- **DMZ** – Public-facing servers, isolated from internal network
- **External/Internet** – The wider world, accessed through the router's gateway

Devices on different subnets can't communicate directly—traffic must route through the appropriate gateway, which may have firewall rules.

### Firewalls
Firewalls control what traffic can pass between network zones:

- Block specific ports (no SSH from outside)
- Block specific IPs or ranges
- Allow only certain services through

Firewalls create obstacles. Finding misconfigured rules, exploiting allowed services, or bypassing them entirely through pivoting are core hacking activities.

## Reaching Your Target

### Direct Access
If your device is on the same network as the target, you can connect directly (assuming no firewall blocks you).

### Pivoting
When a target is unreachable from your position, you **pivot** through intermediate machines:

1. Compromise a machine that *can* reach the target
2. SSH or establish a session on that machine
3. Attack the target from your new position

Example: You're on public WiFi. The target database is on an internal VLAN. You compromise a developer's laptop on the corporate LAN, then pivot through it to reach the database subnet.

### Physical Access
Some targets are air-gapped or otherwise unreachable remotely. The infiltrator must physically connect—plug in a device, insert a USB, or access a terminal directly. The hacker then operates through that physical bridge.

## Trace and Detection

Security isn't passive. The game uses a **combination system** for detection:

### Active Trace
During an intrusion, security systems actively trace your connection:

- A trace timer shows how close they are to finding you
- More aggressive actions (failed logins, noisy scans) speed up the trace
- Disconnect before the trace completes, or face consequences
- **Bouncing** through multiple machines slows the trace by adding hops

### Action-Based Alerts
Specific actions can trigger immediate alerts:

- Failed login attempts
- Port scans detected by IDS
- Accessing honeypot files
- Suspicious traffic patterns

Alerts may dispatch security to investigate, lock down systems, or begin active countermeasures.

### Passive Logging
Even if you escape active detection, your actions leave logs:

- Connection logs on routers and servers
- Authentication records
- File access timestamps

If you don't cover your tracks, investigators may trace the attack back to you later—that cafe you hacked from, the bounce points you used, the trail of compromised machines.

## Services and Access

Devices expose **services** that can be interacted with:

| Service | Purpose | Attack Surface |
|---------|---------|----------------|
| SSH (22) | Remote shell access | Brute force, key theft, exploits |
| HTTP (80/443) | Web interfaces | Web app vulnerabilities, default creds |
| SMB (445) | File shares | Credential reuse, lateral movement |
| Database ports | Data storage | SQL injection, credential theft |
| Camera feeds | Surveillance | View/disable to help infiltrator |

Not every device runs every service. Part of reconnaissance is discovering what's available.

## DNS and InterNIC

### DNS Resolution
Devices use DNS servers to resolve hostnames to IPs. The configured DNS server matters—different DNS servers may have different records, and compromising or spoofing DNS opens attack possibilities.

### InterNIC
For internet targets, **InterNIC** serves as a public registry—a phonebook of known servers and their addresses. Some targets are listed publicly. Others must be discovered through intel, logs, or social engineering.

## Environmental Factors

The physical world affects network operations:

- **Power outages** – Devices lose power, connections drop (including yours if you're in the affected area)
- **Network congestion** – Latency increases, operations take longer
- **Signal strength** – Wireless connections may be unreliable at range
- **Physical destruction** – Cut cables, destroyed routers, disabled infrastructure

These create opportunities for sabotage—and risks for the unwary hacker.
